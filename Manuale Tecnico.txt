Manuale Tecnico - Gestionale Magazzino Didattico Firebase-only (v1.42.0)

Scopo della release
===================
La versione 1.42.0 mantiene il refactoring Firebase-only e avvia il refactoring di dominio dell'area Vendite. Firestore resta l'unica persistenza dei dati gestionali; il browser mantiene solo cache di sessione in memoria. La logica più critica di ordini cliente, DDT cliente e fatture è stata separata dalla UI per ridurre fragilità e regressioni.

Architettura
============
- Entry point: index.html carica src/main.js come ES Module.
- Core applicativo: src/core/app.js crea App, event bus, db wrapper, home, stats e riferimenti Firebase.
- Dati: src/core/db.js normalizza e mantiene lo stato in memoria, poi sincronizza su Firestore tramite src/core/firestoreRepo.js.
- Schema: src/core/dbSchema.js definisce struttura iniziale e normalizzazione.
- Backup: src/domain/backupMapper.js converte backup JSON precedenti nello schema corrente.
- Vendite: src/domain/sales.service.js contiene calcoli e mutazioni per ordini cliente, DDT cliente, fatture e rollback.
- Feature: src/features/* contiene le aree funzionali caricate dal bootstrap modulare; src/features/vendite/index.js resta focalizzato su DOM, modali, toast, stampa e orchestrazione.

Persistenza e bootstrap
=======================
1. App.boot() inizializza Firebase.
2. Prima del login non viene caricato alcun archivio applicativo locale.
3. Dopo login Firebase, App.db.loadFromFirebase() legge Firestore e popola la cache di sessione.
4. App.db.save(db) aggiorna la cache di sessione e accoda la sincronizzazione Firestore.
5. App.db.ensure() fallisce se i dati non sono ancora stati caricati da Firebase.

Regola tecnica: nessuna funzione applicativa deve leggere o scrivere su localStorage o sessionStorage. Non sono previsti usi ammessi di storage browser.

Tema e identificativi browser
=============================
Il tema UI segue la preferenza di sistema al caricamento della pagina. Il toggle tema resta funzionante nella sessione corrente, ma non viene salvato nel browser. Non viene generato alcun device id persistente: l’identità applicativa dipende dall’utente Firebase autenticato.

Backup/import/export
====================
- Export: sincronizza prima la cache corrente, rilegge Firestore e scarica un backup JSON remoto.
- Import: normalizza il JSON, opzionalmente svuota lo spazio Firestore utente e scrive su Firebase.
- Dataset esempio: resta una sorgente didattica di anteprima; l'applicazione avviene sempre su Firestore.

Verifica Node.js
================
Eseguire:

npm run verify

Lo script controlla:
- sintassi JavaScript con node --check;
- versione package/release allineata a 1.42.0;
- presenza della voce corrente in CHANGELOG.md;
- test Node.js di dominio con node --test sui flussi principali di vendita;
- assenza della cartella legacy;
- assenza di chiamate operative alla vecchia modalità locale;
- assenza totale di localStorage e sessionStorage.

Checklist manuale prima della consegna
======================================
1. Avvio con server statico, non da file://.
2. Login Firebase valido.
3. Caricamento Firestore riuscito.
4. Creazione/modifica di un dato semplice e sync senza errore.
5. Export backup Firebase.
6. Import dataset esempio su account di test.
7. Sezione Release e changelog visibile con versione 1.42.0.
8. Verifica manuale di ordine cliente, DDT cliente, cancellazione DDT, fattura e cancellazione fattura su account di test.

Note sulla patch badge/tooltip
==============================
Il file script_badges_tooltips_patch.js resta incluso come patch UI indipendente per badge colorati e tooltip sugli stati documentali. Non gestisce persistenza dati e non partecipa al refactoring Firebase-only.

Aggiornamento 1.43.0: la logica di dominio dell’area Acquisti/Fornitori è stata separata in src/domain/purchasing.service.js. La feature acquisti mantiene UI, rendering e orchestrazione, mentre stato ordine, DDT, quarantena, resi e rollback sono coperti da test Node.js.

Aggiornamento 1.44.0: backup, import, export e cancellazione dati Firebase sono ora orchestrati da src/domain/backup.service.js. La schermata Impostazioni mantiene le sole responsabilità UI e invoca il servizio per normalizzazione, controlli di allineamento e operazioni Firestore.

Versione 1.56.0 - Mutazioni DB controllate estese

- I moduli feature usano App.db.mutate(label, updater) per modificare il database applicativo.
- App.db.save() resta confinato al wrapper dati e ai percorsi infrastrutturali, evitando salvataggi diretti sparsi nelle schermate.
- I flussi documentali di vendite e acquisti applicano modifiche documento e variazioni stock/quarantena nello stesso perimetro di mutazione applicativa.
- La verifica struttura intercetta regressioni su App.db.save() nei moduli feature.

Versione 1.57.0 - Mutazioni DB transazionali
- Aggiunto il runner src/core/dbMutation.js.
- App.db.mutate(label, updater) applica le modifiche su una copia draft dello stato in memoria.
- Se l'updater genera errore, lo stato condiviso non viene salvato e non viene sincronizzato.
- Sono vietate mutazioni annidate e mutazioni asincrone per ridurre il rischio di stati parziali.
- Le note della home usano ora db.mutate(), così non restano salvataggi DB diretti fuori dal wrapper dati.

Versione 1.58.0 - Split UI features
La schermata Vendite è stata alleggerita con moduli UI dedicati ai rendering di ordini cliente, DDT cliente e fatture. La schermata Acquisti è stata alleggerita con moduli UI dedicati a ordini fornitore, DDT/resi e quarantena. La schermata Impostazioni è stata separata in moduli per azienda, utenti, avanzate e release/changelog. Gli entrypoint mantengono il wiring principale e i flussi continuano a usare service di dominio e mutazioni DB controllate.

Versione 1.61.0 - Guardia conflitti Firestore

- In caso di differenza viene generato un errore esplicito e il salvataggio non sovrascrive i dati remoti.
- Aggiunti test Node.js dedicati alla logica di conflitto revisionale.

Versione 1.65.0 - Chiusura refactoring strutturale

Questa versione chiude il refactoring strutturale. Il progetto è Firebase-only, non usa localStorage/sessionStorage per il codice applicativo, separa logica di dominio e UI, centralizza le mutazioni DB e include verifiche Node.js per release, struttura, HTML, stampa, smoke DOM e dominio.
